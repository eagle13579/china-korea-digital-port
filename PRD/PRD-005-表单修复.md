## 1. 产品概述

### 1.1 问题陈述

中韩出海数智港已经完成后端FastAPI API的建设，包含三个完整的表单提交接口：

- `POST /api/v1/contact` — 联系表单提交
- `POST /api/v1/demo` — 演示申请提交
- `POST /api/v1/pricing/inquiry` — 定价咨询提交

前端页面（index.html, pricing.html, team.html）中存在以下问题：

1. **index.html 联系表单**：UI已存在（第228-250行），但表单中的input字段**缺少 name 和 id 属性**，数据采集依赖 placeholder 选择器（main.js 第36-41行），存在风险。
2. **main.js 表单提交逻辑**：已实现 `fetch('/api/v1/contact')` 提交（第58行），但字段名使用 `contact` 而非后端期望的 `name`，后端通过 `ContactForm.validate_contact` 做了兼容处理，不够健壮。
3. **pricing.html 定价页面**：当前没有任何咨询表单，仅有 `index.html#contact` 链接跳转。作为独立的定价页面，缺少即时咨询的入口。
4. **team.html 数字员工页面**：`inviteEmployee()` 函数（第813行）仅用 `confirm()` + 页面跳转，没有真正的表单提交流程。
5. **表单提交反馈**：所有表单目前均使用 `alert()` 弹窗提示，用户体验差。
6. **字段名一致性**：前端字段名与后端 Pydantic 模型字段名未经严格校验，存在 `contact vs name` 等映射差异。

**核心问题**：表单虽然能工作，但存在字段名不一致、反馈体验差、部分表单缺失、双向数据验证不完善等问题，影响转化率和用户信任度。

### 1.2 产品愿景

让所有表单的提交链路变为：**前端填写 → 字段名一致校验 → 调用后端API → Toast消息反馈 → 数据落库 → 用户感知完成**。用户在任何页面提交任何表单时，都能得到即时、友好、中韩双语的反馈，同时确保数据的完整性和一致性。

### 1.3 目标用户

| 用户类型 | 描述 |
|---------|------|
| 韩企潜在客户 | 填写联系/演示/咨询表单的企业决策者，需要得到即时确认反馈 |
| 中国合作伙伴 | 通过网站了解服务并提交需求的中间人角色 |
| 运营团队 | 管理员在后台查看提交数据，需要字段完整、可读 |

### 1.4 成功指标（KPI）

| 指标 | 目标值 | 衡量方式 |
|------|--------|---------|
| 表单提交成功率 | >= 95% | 后端日志统计 |
| 字段名一致性命中率 | 100%（零 validation error） | API 422错误计数 |
| Toast消息可见率 | 100%（所有提交都触发反馈） | 前端事件埋点 |
| 从表单提交到成功确认的时间 | <= 1.5s | Lighthouse / 性能日志 |
| alert弹窗使用率 | 0%（完全替换为Toast） | 代码搜索 audit |

## 2. 功能规格

### 2.1 用户故事

#### US-001：提交联系表单并获得即时反馈

**作为** 一位访问网站的企业负责人
**我想要** 在填写完联系表单后，点击「提交咨询」按钮，看到友好的成功/失败提示
**以便于** 确认我的咨询已经成功送达，不用怀疑是否提交成功

**验收条件：**
- 所有必填字段（企业名称、联系人、电话、邮箱）有客户端校验
- 提交时按钮显示「提交中...」并禁用，防止重复提交
- 提交成功后显示绿色Toast消息，中文版：「感谢您的咨询！我们将在24小时内与您联系。」
- 提交失败后显示红色Toast消息，中文版：「提交失败，请稍后重试。」并保留已填数据
- 提交成功后自动清空表单
- Toast消息3.5秒后自动消失，支持手动关闭

#### US-002：提交演示申请

**作为** 一位想预约产品演示的潜在客户
**我想要** 在单独的演示申请页面或弹窗中填写期望日期和备注信息
**以便于** 预约一个方便的时间点进行产品演示

**验收条件：**
- 演示表单可以添加到 index.html 的联系表单区域下方（作为独立弹窗或选项卡）
- 必填字段：姓名、企业名称、邮箱
- 可选字段：电话、期望日期、备注
- 提交成功后显示Toast消息：「演示预约已提交！我们的团队将尽快与您确认具体时间。」
- 支持中韩双语Toast

#### US-003：在定价页面直接咨询

**作为** 一位浏览定价页面的潜在客户
**我想要** 在不跳转到首页的情况下，直接提交我对某个套餐的咨询
**以便于** 快速了解我感兴趣的方案细节

**验收条件：**
- pricing.html 增加咨询表单区域（底部或侧边弹窗）
- 表单预填用户正在查看的套餐类型（plan_type）
- 提交到 `POST /api/v1/pricing/inquiry`
- 成功后显示Toast：「您的咨询已收到！销售团队将主动联系您。」

#### US-004：在数字员工页面提交服务邀请

**作为** 一位浏览数字员工的韩企CEO
**我想要** 在点击「邀请Ta为我服务」后，弹出一个表单而不是直接跳转
**以便于** 在页面上留名留联系方式，销售团队据此联系我

**验收条件：**
- 点击「邀请Ta为我服务」打开一个轻量弹窗表单
- 弹窗显示用户选择的数字员工名称（只读字段）
- 表单字段：姓名、公司、邮箱、电话、需求描述
- 提交到后端API `/api/v1/service-inquiry`（需新增）或复用 `/api/v1/contact`
- 提交成功后显示Toast消息

### 2.2 界面描述

#### 2.2.1 Toast消息组件设计

Toast是一个全局的消息提示组件，出现在页面右上角，支持成功/错误/信息三种类型。

```
┌──────────────────────────────────┐
│  ✅ 感谢您的咨询！          [✕]  │  ← 绿色（成功）
│  我们将在24小时内与您联系。      │
└──────────────────────────────────┘

┌──────────────────────────────────┐
│  ❌ 提交失败，请稍后重试。   [✕]  │  ← 红色（错误）
│  网络连接异常                  │
└──────────────────────────────────┘
```

**Toast 规范：**

| 属性 | 值 |
|------|-----|
| 定位 | fixed, top: 20px, right: 20px |
| z-index | 9999（最高层） |
| 最大宽度 | 420px |
| 内边距 | 16px 20px |
| 边框半径 | 12px |
| 背景 | success: #16a34a（深色）/ #dcfce7（浅色）
|       | error: #dc2626（深色）/ #fef2f2（浅色）
| 图标 | 成功: bi-check-circle-fill, 错误: bi-x-circle-fill |
| 关闭按钮 | 右上角 bi-x，点击关闭 |
| 自动关闭 | 3.5秒后自动淡出 |
| 动画 | 从右侧滑入（translateX(100%) → 0），opacity 0 → 1 |
| 堆叠 | 多个Toast垂直堆叠，间距 12px |

#### 2.2.2 联系表单字段修正（index.html）

当前表单input缺少 `name` 和 `id` 属性，修复后：

```html
<input type="text" class="form__input" name="company" id="form_company"
       placeholder="请输入企业名称" ...>
<input type="text" class="form__input" name="contact" id="form_contact"
       placeholder="请输入联系人姓名" ...>
<!-- name属性值 = 后端字段名，确保一致性 -->
```

前端提交时使用 `name` 属性自动收集数据，而非placeholder选择器。

## 3. 验收标准

### 3.1 产品层验收

#### P-AC-001：联系表单完整提交链路

- **Given** 用户在index.html的联系表单填写了所有必填字段
- **When** 用户点击「提交咨询」按钮
- **Then** 按钮显示「提交中...」并禁用
- **And** 数据以JSON格式POST到 `/api/v1/contact`
- **And** 字段名完全匹配：`name, company, email, phone, message`
- **And** 成功后显示绿色Toast：「感谢您的咨询！我们将在24小时内与您联系。」
- **And** Toast自动在3.5秒后消失
- **And** 表单自动重置为空

#### P-AC-002：字段验证与错误反馈

- **Given** 用户未填写必填字段就提交表单
- **When** 点击提交按钮
- **Then** 客户端校验阻止提交，在对应字段下方显示红色校验提示
- **And** 不发起API请求

#### P-AC-003：网络错误时的降级处理

- **Given** 用户提交表单但网络异常
- **When** API请求失败
- **Then** 显示红色错误Toast：「提交失败：网络连接异常，请稍后重试。」
- **And** 表单数据保留，用户可再次尝试
- **And** 按钮恢复为可点击状态

#### P-AC-004：定价页咨询表单

- **Given** 用户在 pricing.html 浏览定价方案
- **When** 用户点击「立即咨询」或「免费咨询」按钮
- **Then** 在当前页面弹出咨询表单弹窗（或滚动到表单区域）
- **And** 表单预填 `plan_type`（免费初评/深度方案/年订阅）
- **And** 提交到 `/api/v1/pricing/inquiry`
- **And** 成功后显示Toast：「您的咨询已收到！销售团队将主动联系您。」

#### P-AC-005：数字员工邀请表单

- **Given** 用户在 team.html 页面浏览数字员工
- **When** 用户点击某位数字员工的「邀请Ta为我服务」按钮
- **Then** 打开一个弹窗表单
- **And** 表单中自动显示该员工的姓名（只读）
- **And** 用户填写自己的姓名、公司、邮箱、电话、需求
- **And** 提交到 `/api/v1/service-inquiry`
- **And** 成功后显示Toast：「邀请已提交！销售团队将主动联系您。」

#### P-AC-006：中韩双语Toast

- **Given** 用户当前语言设置为韩语
- **When** 任何表单提交成功或失败
- **Then** Toast消息显示韩语文案
- **And** Toast的关闭按钮不影响语言切换

### 3.2 技术层验收

#### T-AC-001：字段名一致性

- 前端所有表单字段名与后端Pydantic模型字段名逐一对应
- 移除 `contact` 字段的依赖，统一使用 `name`
- 运行 `curl -X POST -H "Content-Type: application/json" -d '{"name":"test"}' http://localhost:8000/api/v1/contact` 返回200而非422

#### T-AC-002：Toast组件独立可复用

- Toast组件封装为独立函数 `showToast(message, type)` 
- 不依赖任何第三方库
- 支持堆叠显示多个Toast
- 支持通过 `data-lang-key` 属性自动获取当前语言的文案

#### T-AC-003：性能

- Toast组件的DOM插入和移除无布局抖动（使用 `transform` 而非 `left/top`）
- 表单提交到Toast显示的时间 <= 300ms
- 不使用任何外部依赖（纯原生JS）

#### T-AC-004：安全

- 所有输入在客户端进行XSS过滤（`textContent` 而非 `innerHTML`）
- API请求使用 POST + JSON Content-Type
- 按钮防重复提交（disabled状态）

#### T-AC-005：向后兼容

- 现有 `contact` 字段兼容保留3个月过渡期（后端 `validate_contact` 不改）
- 所有表单不破坏现有的语言切换逻辑
- 不破坏现有的主题切换逻辑

## 4. 技术方案

### 4.1 总体架构

```
用户填写表单
    │
    ▼
客户端校验（必填字段 + 格式校验）
    │
    ▼ (校验通过)
收集表单数据（通过 input[name] 自动收集）
    │
    ▼
字段名映射校验（前端字段 ↔ 后端字段一对一对齐）
    │
    ▼
POST /api/v1/{endpoint} (fetch JSON)
    │
    ├── 成功 → showToast(successMsg, 'success') → 清空表单
    │
    └── 失败 → showToast(errorMsg, 'error') → 保留数据
```

### 4.2 Toast组件实现

```javascript
// toast.js — 全局Toast消息组件
function showToast(message, type = 'success', duration = 3500) {
    const container = document.getElementById('toast-container');
    if (!container) {
        const div = document.createElement('div');
        div.id = 'toast-container';
        div.style.cssText = 'position:fixed;top:20px;right:20px;z-index:9999;display:flex;flex-direction:column;gap:12px;';
        document.body.appendChild(div);
    }
    
    const toast = document.createElement('div');
    const icon = type === 'success' ? 'bi-check-circle-fill' : 'bi-x-circle-fill';
    const bgColor = type === 'success' 
        ? 'var(--toast-success-bg, #16a34a)' 
        : 'var(--toast-error-bg, #dc2626)';
    
    toast.style.cssText = `...`; // 见样式规范
    toast.innerHTML = `
        <i class="bi ${icon}"></i>
        <span>${message}</span>
        <i class="bi bi-x" style="cursor:pointer;"></i>
    `;
    
    document.getElementById('toast-container').appendChild(toast);
    
    // 自动移除
    setTimeout(() => {
        toast.style.opacity = '0';
        toast.style.transform = 'translateX(100%)';
        setTimeout(() => toast.remove(), 300);
    }, duration);
}
```

### 4.3 表单数据收集标准化

**当前问题（main.js 第36-41行）：**
```javascript
// 当前：依赖placeholder选择器，脆弱
const formData = {
    company: document.querySelector('input[placeholder="请输入企业名称"]')?.value || '',
    contact: document.querySelector('input[placeholder="请输入联系人姓名"]')?.value || '',
    // ...
};
```

**修复后：**
```javascript
// 推荐：通过 input name 属性自动收集
const form = document.querySelector('.form');
const formData = {};
form.querySelectorAll('input, textarea').forEach(el => {
    if (el.name) {
        formData[el.name] = el.value;
    }
});
// 同时为HTML添加 name 属性
```

## 附录 A：字段名一致性检查脚本（参考）

可在 CI 或 pre-commit hook 中加入以下检查逻辑：

```bash
#!/bin/bash
# check_field_consistency.sh
# 扫描前端表单字段名与后端 Pydantic 模型的字段名是否一致

# 前端字段名提取（从所有HTML文件中提取 name 属性值）
FRONTEND_FIELDS=$(grep -roh 'name="[^"]*"' *.html 2>/dev/null | sort -u)

# 后端字段名提取（从 models.py 中提取 Field 定义）
BACKEND_FIELDS=$(grep -oh '^\s\+[a-z_]\+\s*:' backend/models.py | sed 's/://' | sed 's/^\s\+//' | sort -u)

# 对比
echo "=== 前端字段名 ==="
echo "$FRONTEND_FIELDS"
echo "=== 后端字段名 ==="
echo "$BACKEND_FIELDS"
echo "=== 差异（可能在对应文件中存在的字段名）==="
diff <(echo "$FRONTEND_FIELDS") <(echo "$BACKEND_FIELDS")
```
