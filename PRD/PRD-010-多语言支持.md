## 1. 产品概述

### 1.1 一句话定位
实现中韩英三语切换，覆盖中文母语用户、韩语母语用户和英语通用用户，让网站对三类用户群体都友好可读。

### 1.2 目标用户
| 用户类型 | 描述 |
|---------|------|
| 韩企CEO/合规负责人 | 韩语母语，需要韩文界面 |
| 中国合作伙伴 | 中文母语，默认使用中文 |
| 在韩外籍人士/国际企业 | 英语作为工作语言，需要英文界面 |
| 海外投资者/咨询公司 | 英语用户，了解中韩出海服务 |

### 1.3 痛点分析
**当前语言能力（来自代码审计）：**
- 已有中韩双语框架 ✅
  - `data-lang-key` 属性在 HTML 中广泛使用 ✅
  - `translations` 对象在 main.js 中定义中韩双语 ✅
  - 语言切换按钮（中文 / 한국어）已在所有页面 ✅
  - 语言偏好通过 `localStorage.getItem('lang')` 持久化 ✅
  - team.html 有 `getEmployeeField()` 中韩字段映射函数 ✅
  - main.js 有 `updateSEOMeta(lang)` 函数支持SEO标签切换 ✅
- **当前不足：**
  1. 缺少英语（en）语言支持
  2. 语言切换仅在客户端JS层面，后端API返回的文案（如Toast消息、定价数据）均为中文写死
  3. 部分页面（如 pricing.html FAQ）文案硬编码在HTML中，未走 data-lang-key 体系
  4. hreflang 标签只包含 zh-CN 和 ko-KR，缺少 en
  5. 语言切换按钮为两个，需要增加为三个

### 1.4 北极星关联
| 北极星维度 | 预期健康度提升 | 说明 |
|:----------|:--------------:|:-----|
| C商业化 | +3% | 英语支持覆盖国际用户群体 |
| T团队 | +2% | 统一的多语言框架减少维护成本 |

## 2. 功能规格

### 2.1 用户故事

#### US-001：三语切换
**作为** 任何语言背景的用户
**我想要** 在网站右上角看到三个语言切换按钮（中文 / 한국어 / English）
**以便于** 切换到我最舒服的语言浏览网站

**验收条件：**
- 语言切换按钮从2个增加到3个
- 默认语言根据浏览器设置自动识别（优先：zh-CN → ko → en）
- 语言选择通过 localStorage 持久化
- 切换后整个页面（包括导航、Hero、服务、定价、数字员工、表单、Footer、Toast）全部切换
- 切换过程无页面刷新或白屏闪烁

#### US-002：英文SEO
**作为** 英语用户的搜索引擎
**我想要** 英文版页面被Google正常索引
**以便于** 英语用户通过Google搜索发现网站

**验收条件：**
- `hreflang` 标签增加 `en` 版本
- `<html lang="en">` 在英文模式下正确设置
- `meta description` 和 `og:title` 有英文版本
- JSON-LD 结构化数据包含英文名称和描述
- sitemap.xml 增加英文URL

#### US-003：后端API多语言
**作为** 英语用户
**我想要** 提交表单后看到的Toast消息也是英文
**以便于** 我能理解系统的反馈

**验收条件：**
- Toast 消息支持 `en` 语言 ✅（当前toast.js第115-136行只支持zh-CN和ko-KR，需新增en）
- 后端API返回的 `message` 字段支持根据请求头 `Accept-Language` 返回对应语言
- 定价API返回的数据包含英文name和description

### 2.2 界面描述

#### 2.2.1 语言切换按钮

当前：
```html
<button class="lang-btn" data-lang="zh-CN">中文</button>
<button class="lang-btn" data-lang="ko-KR">한국어</button>
```

改为：
```html
<button class="lang-btn" data-lang="zh-CN">中文</button>
<button class="lang-btn" data-lang="ko-KR">한국어</button>
<button class="lang-btn" data-lang="en">English</button>
```

#### 2.2.2 翻译数据结构扩展

当前 translations 结构（main.js）：
```javascript
var translations = {
    'zh-CN': { ... },
    'ko-KR': { ... }
};
```

改为：
```javascript
var translations = {
    'zh-CN': { ... },
    'ko-KR': { ... },
    'en': { ... }  // 新增英文版
};
```

## 3. 验收标准

### 3.1 产品层验收

#### P-AC-001：三语切换
- **Given** 用户访问网站
- **When** 点击「English」按钮
- **Then** 所有可见文本切换到英文
- **And** URL不变化（纯前端切换）
- **And** 切换过程无白屏/闪屏

#### P-AC-002：语言持久化
- **Given** 用户切换到英文
- **When** 刷新页面或打开新页面
- **Then** 页面默认显示英文
- **And** 不需要重新选择语言

#### P-AC-003：表单和Toast英文
- **Given** 用户在英文模式下
- **When** 提交任何表单
- **Then** Toast消息显示英文
- **And** 表单校验提示也显示英文

#### P-AC-004：数字员工英文
- **Given** 用户切换到英文
- **When** 查看 team.html
- **Then** 数字员工的头衔、能力、经验显示英文
- **And** 法律免责声明显示英文

### 3.2 技术层验收

#### T-AC-001：性能
- 语言切换不产生额外网络请求（纯前端切换）
- translations 对象内存占用 ≤ 20KB
- 切换时间 ≤ 100ms（无DOM重渲染延迟）

#### T-AC-002：SEO
- 英文版 `<html lang="en">` 正确设置
- hreflang 标签包含 `en` 版本
- sitemap.xml 包含英文版URL

#### T-AC-003：向后兼容
- 现有 data-lang-key 机制不破坏
- 现有语言切换按钮行为不改变
- 现有 localStorage 存储结构兼容（'lang' key）

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 部分文案漏翻译（硬编码在HTML中未走 data-lang-key） | 中 | 中 | 建立「全站文案清单」，逐页扫描硬编码文案 |
| 英文翻译质量（非母语者写的英文不地道） | 中 | 中 | 英文翻译需英语母语者审校 |
| SEO 可能因三语内容重复被惩罚 | 低 | 低 | 正确使用 hreflang 和 canonical 标签，Google 认可多语言 |
| 后端API多语言支持工作量比预期大 | 中 | 中 | MVP阶段先做前端三语，后端API多语言放v2 |

## 8. 不做清单

| 事项 | 原因 | 后续可能 |
|------|------|---------|
| ❌ 日语/越南语等更多语言 | 仅覆盖核心用户群体（中/韩/英） | 有东南亚业务时考虑 |
| ❌ 后端API全量多语言 | MVP先做前端三语，后端API返回固定中文 | v2后端支持 |
| ❌ AI自动翻译 | 质量不可控，需人工翻译 | 不考虑 |
| ❌ 按IP自动识别语言 | 需要IP库，且韩企用户可能在国内访问 | 根据浏览器语言设置自动识别即可 |
| ❌ 翻译管理后台（运营自助编辑） | 当前翻译数量少，直接改代码即可 | v3可考虑 |
| ❌ 图片/海报的多语言版本 | 设计成本高 | v2按需制作 |
